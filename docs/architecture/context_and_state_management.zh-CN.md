# 上下文与状态管理

FinSight-Agent 的多轮能力不是把聊天历史简单拼进提示词，而是把研究状态、运行产物和证据边界结构化保存。本文说明系统如何处理上下文、范围变化、产物复用和已保存运行检查。

## 为什么不能只拼接历史对话

金融研究中的多轮追问通常会改变范围或证据需求。例如：

- “只保留 NVDA，不要继续分析 AMD。”
- “刚才的结论里只展开资本开支。”
- “不要用市场数据，只看披露文件。”
- “这条供应链关系有没有证据支持？”

如果只是把上一轮回答拼进新提示词，模型可能继续使用旧公司、旧范围或旧证据。FinSight-Agent 使用结构化状态来避免这种问题。

## 状态分层

| 状态层 | 保存内容 | 用途 |
| --- | --- | --- |
| 会话状态 | 用户目标、当前问题、当前研究范围、上一轮摘要 | 判断追问是否继承、收缩或重置范围 |
| 图运行状态 | 每个节点的输入、输出、状态、错误和可恢复点 | 支持节点级审计和恢复 |
| 证据状态 | 检索到的上下文行、市场行、行业行、关系行 | 控制下游只能读取限定证据 |
| 数值状态 | 精确数值台账行、单位、期间、来源对象 | 支撑查数和数值校验 |
| 缺口状态 | 来源缺口、指标缺口、关系确认缺口、市场字段缺口 | 决定二次检索或有边界回答 |
| 专家状态 | 专家结论卡、反证、冲突、缺口请求 | 汇总成研究提纲 |
| 写作状态 | 判断提纲、备忘录草稿、校验报告和最终答案 | 支持呈现、修复和后续追问 |
| 产物引用 | 运行目录、摘要文件、台账文件、报告文件 | 支持已保存运行检查 |

## 单轮运行中的状态流

一次完整运行中，状态会按顺序累积：

```mermaid
flowchart LR
  A["用户问题"] --> B["研究计划"]
  B --> C["公司范围"]
  C --> D["证据与工具台账"]
  D --> E["覆盖与缺口"]
  E --> F["专家结论卡"]
  F --> G["判断提纲"]
  G --> H["备忘录草稿"]
  H --> I["校验报告"]
  I --> J["最终答案和会话状态"]
```

每一步都可以单独检查。比如，最终答案出错时，可以回看是研究负责人范围错了、检索证据不足、专家结论卡太薄，还是备忘录写作越界。

## 多轮追问如何处理

多轮追问会先进入上下文判断。系统会比较新问题和上一轮状态：

| 追问类型 | 处理方式 |
| --- | --- |
| 缩小公司范围 | 保留新范围内的证据，失效被排除公司的下游结论 |
| 增加公司或行业 | 回到范围构建和证据执行节点补充检索 |
| 只改输出形式 | 尽量复用已有判断提纲和证据，重新呈现或改写 |
| 查询某个数值 | 优先查数值台账或结构化对象，不重跑完整备忘录 |
| 质疑某条结论 | 回看证据引用和校验报告，必要时要求有边界回答 |
| 更换来源限制 | 失效不符合新来源限制的证据和下游结论 |

这让系统既能复用已有产物，又不会把旧范围带入新问题。

## 失效和复用规则

| 变化 | 可复用 | 必须失效 |
| --- | --- | --- |
| 只要求更短回答 | 已验证判断提纲、证据引用 | 最终呈现文本 |
| 只保留某家公司 | 该公司的证据、数值台账、来源缺口 | 被排除公司的专家结论和备忘录段落 |
| 新增供应链视角 | 原目标公司证据 | 公司范围计划、关系证据、行业/供应链专家输出 |
| 禁用市场数据 | 披露文件和数值台账 | 市场专家输出、市场相关结论 |
| 改成精确查数 | 数值台账和结构化对象 | 复杂备忘录写作路径 |
| 发现关键缺口 | 已支持结论 | 包装过满的完整论点 |

## 已保存运行检查

系统可以读取之前的运行产物，而不是每次都重新检索。已保存运行通常包括：

- 研究计划和节点轨迹。
- 检索结果和工具台账。
- 数值台账。
- 覆盖报告。
- 专家结论卡。
- 备忘录和校验报告。
- 最终答案。

用户可以要求查看某次运行的状态、证据、覆盖缺口或最终答案。对于这种请求，系统应优先读取运行产物，不应无意义地重新跑完整链路。

## 上下文安全边界

上下文管理最重要的是防止旧状态污染新问题：

- 用户明确排除的公司不能继续作为主分析对象。
- 上一轮允许的数据源不能覆盖新一轮的数据源限制。
- 关系假设不能因为在上一轮出现过，就在下一轮升级为事实。
- 旧市场快照不能被包装成实时行情。
- 旧备忘录中的未经支持观点不能作为新一轮证据。

系统通过范围计划、来源边界、产物引用和校验报告共同约束这些问题。

## 当前能力

当前系统已经支持：

- 会话状态保存。
- 已保存运行读取。
- 多轮范围收缩和范围覆盖检查。
- 运行产物引用和状态摘要。
- 基于节点状态的恢复和检查。
- 多轮评测中验证 stale ticker 不应继续污染新问题。

## 当前边界

- 当前会话存储更适合本地开发、演示和诊断评测，多用户生产服务还需要数据库、队列、权限和更强的并发治理。
- 复杂多轮研究中，如果用户连续改变公司、来源、时间和输出形式，仍需要更细的失效策略。
- 上下文可以防止明显越界，但不能弥补上游数据缺口；缺少证据时仍应输出缺口或有边界回答。
