# 多智能体协作机制

本文说明 FinSight-Agent 的多智能体如何协作。当前系统不是多个智能体自由聊天，也不是让多个模型互相辩论后投票；它是一个有权限、有状态、有条件回路的研究工作流。

## 协作方式

多智能体之间通过共享状态和结构化产物通信。每个智能体读取自己被允许看到的输入，产出固定结构的结果，下游节点再继续处理。

典型通信产物包括：

- 研究计划
- 公司范围计划
- 证据需求计划
- 工具观察结果
- 覆盖报告
- 专家结论卡
- 缺口请求
- 判断提纲
- 备忘录草稿
- 校验报告

这种方式比“智能体互聊”更适合金融研究：它能保留证据边界、工具权限、调用成本和失败原因。

## 智能体角色

| 角色 | 主要职责 | 权限边界 |
| --- | --- | --- |
| 研究负责人 | 判断问题类型、研究深度、公司范围、证据需求和激活策略 | 只提出研究需求，不直接检索原始数据 |
| 公司范围与关系构建 | 从公司清单和关系证据中选择候选公司，说明纳入和排除理由 | 关系证据只支持研究范围和假设 |
| 证据执行器 | 调用受控工具检索披露、市场、行业、关系和数值台账 | 只能按计划和预算执行工具 |
| 覆盖检查 | 判断证据是否足够，是否需要二次检索或有边界回答 | 不写最终观点 |
| 基本面专家 | 从披露和数值台账中形成经营与财务判断 | 不调用工具，不使用市场/行业替代公司事实 |
| 市场专家 | 从市场快照中分析市场反应和估值语境 | 不生成实时行情、价格目标或投资建议 |
| 行业/供应链专家 | 解释行业变量、关系假设和经济传导路径 | 不能把假设关系写成已确认客户或合同事实 |
| 风险专家 | 识别反证、冲突、下行情景和证据不足 | 不重复支持性论点，不新增事实 |
| 判断汇总器 | 汇总支持结论、反证和缺口，形成研究提纲 | 不引入专家没有提供的新事实 |
| 备忘录写作器 | 根据已验证提纲写用户可读的研究备忘录 | 不检索、不读原始证据、不新增公司和数值 |
| 校验器 | 检查来源、数值、关系假设和结论强度 | 只做检查和定点修复，不增加分析深度 |
| 呈现器 | 展示答案、证据边界、假设级别和缺口请求 | 不改变结论 |

## 激活策略

系统不会每个问题都启动所有智能体。研究负责人会先判断问题类型，再决定哪些角色必须启动、哪些条件启动、哪些禁止启动。

| 问题类型 | 典型路径 |
| --- | --- |
| 精确查询 | 数值台账或结构化对象优先，通常不启动专家和备忘录写作 |
| 聚焦回答 | 研究负责人、证据执行器、覆盖检查、备忘录写作器和校验器 |
| 标准备忘录 | 基本面、市场、风险专家通常参与；行业专家按问题需要启动 |
| 行业深度 | 公司范围与关系构建、行业/供应链专家、基本面专家、风险专家共同参与 |
| 供应链传导 | 必须检查关系证据、行业证据和公司自身披露边界 |
| 市场分歧 | 市场专家和基本面专家共同解释基本面与市场反应是否一致 |
| 多轮追问 | 先读取上一轮状态，再决定复用、收缩范围还是重新检索 |
| 已保存运行检查 | 读取运行产物和覆盖报告，通常不重新检索 |

## 条件回路

多智能体链路中有两个重要回路。

第一个是覆盖回路。证据执行后，覆盖检查会判断当前证据是否足够。如果缺口是可检索的，系统可以发起二次检索；如果缺口来自当前数据范围之外，则保留为数据边界。

第二个是校验修复回路。备忘录写作后，校验器会检查来源越界、数值漂移、关系假设包装过满或缺口丢失。如果问题可修复，会要求写作器定点修复；如果关键证据不足，会要求答案降级为有边界回答。

两个回路都有预算和停止条件，避免无限重跑。

## 范围决策示例

用户问“英伟达基本面表现如何”时，系统不会简单地只检索英伟达，也不会直接把半导体全行业都拉进来。研究负责人会先判断问题属于哪类：

- 如果用户只问收入、利润率、现金流等自身指标，优先走单公司基本面路径。
- 如果问题提到人工智能资本开支、供应链瓶颈、云厂商需求、出口管制或市场分歧，就进入条件扩展或必须扩展。
- 如果扩展，公司范围节点会按同业、上游供应、下游客户、基础设施依赖、能源传导和非美国供应链线索选择候选公司。

每个纳入公司都需要说明为什么纳入、有哪些来源、关系强度是什么、后续由哪个证据执行器负责。每个排除公司也需要说明排除理由。这样可以让模型利用 603 家公司资产，同时避免盲目搜索全库。

## 缺口如何传递

缺口不是简单的一句“数据不足”，而是结构化请求。常见类型包括：

- 缺少某公司、指标、期间或披露类型的数值台账。
- 缺少估值倍数、事件窗口或市场反应字段。
- 关系链只有假设，没有披露或行业来源确认。
- 当前公司范围不足，需要回到范围构建节点追加公司。
- 投资论点缺少反证，需要风险专家补充。

这些缺口会从专家结论卡进入判断提纲，再进入备忘录和最终呈现。这样用户可以知道哪些结论有证据，哪些只是待验证假设，下一步应补什么。

## 为什么这样设计

金融研究问题经常同时涉及公司事实、行业背景、市场反应和经济关系。如果所有工作都交给一个模型，很容易出现三个问题：

1. 把不同来源的证据混在一起。
2. 把假设写成事实。
3. 在多轮追问中忘记或污染研究范围。

多智能体编排的价值不是“角色越多越好”，而是把职责和权限拆开：谁决策、谁取证、谁分析、谁写作、谁校验，都可以被单独检查。

## 当前边界

- 当前协作是图状态驱动，不是独立智能体服务之间的实时消息总线。
- 专家智能体通常按角色顺序或子图运行，当前重点是质量和可审计性，不是高并发智能体通信。
- 二次检索和修复循环有预算限制；复杂行业深度问题仍可能因为证据不足或成本过高而输出有边界回答。
- 关系图谱主要支持研究范围和传导假设，不能替代外部供应链数据库或已确认合同证据。
