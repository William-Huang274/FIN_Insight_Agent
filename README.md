# FinSight-Agent

[English version](README.en.md)

FinSight-Agent 是一个面向公开公司研究的证据约束型多智能体系统。它的目标不是让大模型凭几段检索文本直接写结论，而是把开放式投研问题拆成一条可检查的研究流程：先判断问题类型和研究范围，再构建公司与产业关系范围，检索公开披露、公司业绩材料、市场快照、行业背景和关系证据，整理可追溯的数值台账，交给不同角色的专家智能体形成结论卡，最后由汇总、写作、校验和呈现节点生成有边界的研究备忘录。

项目当前聚焦美股上市公司研究，但底层方法并不只属于金融场景。凡是需要多源材料、口径约束、证据追溯、工具调用、多轮追问和结论校验的复杂知识工作，都可以复用这套思路。

FinSight-Agent 不做实时行情终端，不做自动交易，也不生成个性化投资建议。它解决的是另一类问题：让大模型参与研究判断和表达，同时保留证据边界、财务口径、产业关系、工具轨迹和质量门控。

## 项目亮点

- **证据先行**：关键结论必须落到来源类型、公司、期间、指标、单位、快照日期或关系证据上，不能只依赖模型自由发挥。
- **多智能体分工**：研究负责人、范围构建、证据执行、基本面、市场、行业、风险、判断汇总、备忘录写作和校验节点各自承担不同职责。
- **公司范围可扩展**：当前扩展资产覆盖 603 家公司，系统可以围绕同业、上游供应、下游客户、基础设施依赖、能源用电传导和非美国供应链线索构建研究范围。
- **数值台账可追溯**：精确查询和财务指标比较优先走结构化台账，记录公司、期间、指标、单位和来源对象，避免模型从长文本中猜数。
- **关系和缺口显式化**：当证据只支持假设、关系未确认、指标缺失或行业来源不足时，系统会把缺口保留下来，而不是把答案包装成完整论点。
- **多轮状态可恢复**：系统保存研究范围、答案、证据引用、节点输出和产物路径，后续追问可以复用已有证据，也可以触发范围收缩或重新检索。
- **工作台可观测**：本地工作台记录运行时间、模型调用量、工具轨迹、节点状态和评测报告，方便做端到端测试、性能分析和质量复盘。
- **可迁移的工程骨架**：这不是单个金融问答脚本，而是一套“范围决策、证据执行、专家判断、缺口升级、结果校验”的可复用研究框架。

## 适合处理的问题

FinSight-Agent 当前适合处理以下任务：

- 单公司基本面分析，例如利润率、资本开支、现金流、研发投入、资产质量或能源公用事业经营指标。
- 多公司横向比较，例如半导体、云厂商、银行、能源、消费、医疗和公用事业公司的财务表现差异。
- 行业深度研究，例如人工智能基础设施、云资本开支、数据中心用电、半导体供应链、银行存款和信用周期。
- 市场反应解释，例如基本面变化、管理层说法、市场快照和估值语境之间是否一致。
- 产业链传导判断，例如某家公司基本面是否需要观察供应商、客户、设备、代工、存储、网络、用电或监管风险。
- 多轮研究追问，例如先生成研究备忘录，再缩小到某几家公司、某个指标、某类证据或某条反证。

## 为什么不是普通检索问答

普通检索问答通常是“检索若干文本，再让模型生成答案”。FinSight-Agent 重点处理金融研究里最容易失控的地方：来源边界、指标口径、产业关系、范围扩展、缺口保留和答案校验。

| 维度 | FinSight-Agent 的做法 |
| --- | --- |
| 问题理解 | 先判断问题类型、研究范围、是否需要扩展公司范围，以及需要哪些证据 |
| 公司范围 | 从公司清单和关系证据中选择候选公司，并记录纳入或排除理由 |
| 工具调用 | 模型只能请求受控工具，真实执行由工具层完成，不能越权读取数据 |
| 检索方式 | 结合关键词检索、对象检索、向量召回、重排、市场快照、行业证据和关系证据 |
| 数值证据 | 使用精确数值台账承载指标、期间、单位和来源，查数问题可绕开大模型 |
| 专家分析 | 专家智能体只消费限定证据，输出可验证的中间结论和缺口请求 |
| 投研主线 | 汇总器保留支持结论和未解决缺口，写作器只基于已验证输入生成备忘录 |
| 质量门控 | 校验器检查越界结论、缺少来源、混用口径、关系假设包装过满等问题 |

## 当前证据范围

公开仓库不包含私有数据、生成索引、模型缓存、运行产物或接口密钥。用户可以用自己的公开披露、市场快照和行业数据生成同类产物，再通过配置文件接入。

| 证据类型 | 支持的分析 | 使用边界 |
| --- | --- | --- |
| 公开披露文件 | 年报、季报中的财务事实、业务描述、风险因素和管理层讨论 | 年度、季度、年初至今和过去十二个月口径不能混用 |
| 公司业绩材料 | 业绩公告、管理层解释、业务动量、经营叙事和指引口径 | 公司发布口径不能替代披露文件中的财务事实 |
| 市场快照 | 离线价格表现、相对收益、事件窗口和估值语境 | 不是实时行情，必须带有快照日期 |
| 行业快照 | 行业主题、供需变化、监管背景、市场结构和产业趋势 | 支持背景判断，不能覆盖公司自身披露事实 |
| 关系图谱 | 同业、供应链、客户、基础设施、能源和地区风险传导 | 关系强度分为已验证、推断、假设和证据缺口 |
| 精确数值台账 | 公司、期间、指标、单位和来源对象绑定 | 没有台账行时必须标记缺口，不能补写数字 |
| 语义检索索引 | 面向长文本和对象证据的语义召回 | 当前主要用于召回增强，不等同于事实校验 |

## 多智能体研究链路

```mermaid
flowchart LR
  A["用户问题"] --> B["研究负责人"]
  B --> C["公司范围与关系构建"]
  B --> D["证据执行器"]
  C --> D
  D --> E["覆盖检查"]
  E --> F["角色化专家"]
  F --> G["判断汇总"]
  G --> H["备忘录写作"]
  H --> I["校验与修复"]
  I --> J["结果呈现与会话状态"]
```

核心节点职责：

| 节点 | 职责 |
| --- | --- |
| 研究负责人 | 判断问题类型、研究范围、证据需求、预算上限，以及是否需要扩展公司范围 |
| 公司范围与关系构建 | 按同业、上游、下游、基础设施、能源和地区线索选择公司，并记录纳入和排除理由 |
| 证据执行器 | 调用检索、数值台账、市场、行业、关系和语义召回工具，生成下游可用证据 |
| 覆盖检查 | 判断证据是否足够支撑任务，明确缺失指标、缺失市场字段或关系确认需求 |
| 角色化专家 | 基本面、市场、行业和风险专家分别产出结论卡，不直接扩大范围 |
| 判断汇总 | 汇总已支持结论，同时保留未解决缺口和反证需求 |
| 备忘录写作 | 在已验证提纲内写研究备忘录，关键缺口阻塞时写有边界的备忘录 |
| 校验与修复 | 检查来源、数值、关系假设、证据缺口和结论强度是否匹配 |
| 结果呈现 | 展示答案、来源边界、假设级别、缺口请求和可继续追问的会话状态 |

## 多智能体如何协作

当前多智能体不是每个角色各自打开一个无限制聊天窗口，也不是让多个模型自由互相辩论。它更接近一个有权限控制的研究工作流：所有智能体通过共享状态、结构化任务卡、工具观察结果、专家结论卡和缺口请求协作。上游节点的输出会成为下游节点的输入；下游节点发现证据不足时，可以触发二次检索、范围重构或有边界回答。

运行时大致分为六步：

1. **研究负责人定任务**：先判断问题属于精确查询、单公司分析、同业比较、行业深度、市场分歧、供应链传导或风险反证，再输出需要启动哪些智能体、允许使用哪些来源、是否需要扩展公司范围和预算上限。
2. **公司范围节点补范围**：如果问题需要产业链或同业视角，范围节点会检查公司清单和关系证据，给出纳入公司、排除公司、关系强度、来源可用性和后续补证要求。
3. **证据执行节点取材料**：真正的检索和查数由受控工具执行，包括披露文件、业绩材料、数值台账、市场快照、行业快照、关系图谱和语义召回。工具调用会写入工具台账，记录谁调用了什么、拿到多少证据、是否产生来源缺口。
4. **覆盖检查决定是否返工**：覆盖检查节点会判断当前证据是否足够。如果缺的是可检索证据，系统可以进入二次检索；如果缺的是当前数据根本没有的来源，就保留为缺口，禁止模型补写。
5. **专家智能体分别读证据**：基本面、市场、行业/供应链、风险专家不直接检索，也不直接改最终答案。它们只读取限定证据，输出结论卡、反证、冲突和缺口请求。
6. **汇总、写作、校验闭环**：判断汇总器把专家结论和未解决缺口合成研究提纲；备忘录写作器只按这个提纲写答案；校验器检查来源、数值、关系假设和结论强度，必要时触发修复或要求最终答案降级为有边界回答。

所以，当前系统是能协作、能“通信”的，但通信方式是受控的：不是任意自然语言互聊，而是通过共享状态里的结构化产物传递信息。典型共享产物包括研究计划、公司范围计划、证据需求、工具观察、来源缺口、专家结论卡、判断提纲、备忘录草稿、校验报告和最终呈现。

## 范围决策与产业关系

复杂问题往往不是“只问哪家公司就只看哪家公司”。例如用户问英伟达基本面表现时，系统需要先判断这是不是单公司基本面问题，还是需要纳入云资本开支、存储、代工、设备、服务器、网络、电力、出口管制和市场反应等外部线索。

研究负责人会先把问题归入若干研究模式：

- 单公司基本面
- 同业比较
- 业绩持续性
- 供应链和客户传导
- 基本面与市场反应分歧
- 行业深度研究
- 风险和反证检查

然后给出范围决策：

- **不扩展**：问题可以由目标公司自身披露和数值台账回答。
- **条件扩展**：需要先检查清单、关系证据或行业来源，再决定是否纳入更多公司。
- **必须扩展**：问题本身涉及供应链、客户、同业或产业传导，只看目标公司会遗漏关键证据。

公司范围构建节点不会盲目搜索全库。每个纳入公司都需要说明纳入理由、可用来源、关系强度和后续负责人；每个排除公司也要说明为什么暂不纳入。这样既能让模型利用新增公司数据，又能防止“把所有公司都搜一遍”的失控行为。

## 缺口升级

FinSight-Agent 不把缺证据当成失败，而是把缺口作为研究结果的一部分。常见缺口包括：

- 某家公司、指标、期间或披露类型缺少数值台账行。
- 市场快照缺少估值倍数、事件窗口或市场反应字段。
- 产业链关系只有假设，没有披露、清单或行业来源确认。
- 当前公司范围不足，需要回到范围构建节点追加候选公司。
- 论点缺少反证，需要风险专家或校验器提出最高价值的反证请求。

最终答案会区分“证据支持的结论”“合理但未确认的假设”“需要补证的缺口”和“不能回答的边界”。这对投研场景很重要，因为很多真实研究问题的价值并不在于立刻写出完整结论，而在于知道下一步该补什么证据。

## 工作台与可观测评测

项目提供本地工作台，用于把数据配置、运行任务、节点轨迹和评测报告集中到一个后端入口。工作台当前支持：

- 导入和检查本地数据配置。
- 启动单轮、多轮和评测任务。
- 记录每个任务的运行时间、工具调用、模型调用量和节点状态。
- 生成可复盘的评测报告。
- 使用常驻工作进程缓存披露清单、重排模型和向量数据库连接，降低冷启动成本。

启动工作台：

```powershell
.\scripts\workbench\run_workbench.ps1
```

只启动后端：

```powershell
python scripts/workbench/start_workbench.py
```

详细说明见 [工作台快速开始](docs/workbench/workbench_quickstart.zh-CN.md)。

## 快速开始

安装依赖：

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

公开仓库可以直接运行结构检查，不需要接口密钥或私有数据：

```powershell
python scripts/eval_context/evaluate_sec_agent_resume_closeout_readiness.py --timeout-s 600
python -m pytest tests/test_resume_closeout_readiness.py tests/test_sec_agent_context_source_policy.py tests/test_market_snapshot_fixture.py
```

更快的本地合同检查：

```powershell
python scripts/eval_context/evaluate_sec_agent_resume_closeout_readiness.py `
  --skip-main-chain-case-suite `
  --skip-context-load-smoke `
  --skip-latency-profile
```

## 完整数据演示

完整链路需要本地已有数据产物、检索索引和模型服务。接口密钥只通过命令行环境变量注入，不写入仓库文件。

```bash
export LLM_BACKEND=deepseek
export MODEL_NAME=deepseek-v4-pro
export API_KEY_ENV=DEEPSEEK_API_KEY
export DEEPSEEK_API_KEY="<set-in-shell-only>"
```

复制配置模板到被 Git 忽略的 `.env`，再替换成本地数据产物路径：

```bash
cp configs/sec_agent_full_source_demo.env.example .env
```

单轮完整数据演示：

```bash
SEC_AGENT_PROFILE_ENV=.env bash scripts/cloud/sec_agent_interactive.sh ask-full-source-api \
"结合 SEC 10-K、最新 10-Q、8-K 业绩新闻稿和最近三个月市场快照，比较 NVDA、AMD、MSFT、AMZN、GOOGL 的 AI 基本面、管理层解释、市场反应和估值分歧。"
```

多轮会话演示：

```bash
SEC_AGENT_PROFILE_ENV=.env bash scripts/cloud/sec_agent_interactive.sh session-full-source-api
```

会话内常用命令：

```text
/state
/context
/answer
/exit
```

## 质量评测

本项目不只检查最终备忘录是否流畅，还检查每个节点是否正确理解任务、合理启动、真实调用工具、消费合法证据、输出可验证中间结果，并控制工具预算和模型调用成本。

当前质量体系覆盖：

- 问题理解与任务授权。
- 公司范围和产业关系构建。
- 证据充分性与来源边界。
- 财务、市场、行业和风险分析。
- 投资论点、反证和观察项。
- 多轮上下文、范围变更和产物复用。
- 输出结构、用户可用性和越界校验。
- 运行时间、模型调用量和工具轨迹。

最新一轮完整链路评测覆盖 20 个真实风格案例，结果全部通过：

| 案例类型 | 通过情况 |
| --- | --- |
| 精确查询 | 2 / 2 |
| 聚焦回答 | 2 / 2 |
| 标准研究备忘录 | 5 / 5 |
| 行业深度研究 | 4 / 4 |
| 多轮追问 | 4 / 4 |
| 范围决策和缺口升级 | 3 / 3 |

这轮评测说明扩展公司范围、产业关系构建、语义召回、专家缺口升级、备忘录写作和结果呈现已经形成可用闭环。平均单题耗时约 155 秒，最慢案例约 331 秒；总模型调用量约 90.6 万 token。这个结果可以视为后端诊断门控通过，但还不能等同于生产级服务性能。

## 当前边界

- 公开仓库不包含私有披露文件、市场数据、生成索引、运行输出、模型缓存或接口密钥。
- 市场快照是离线数据，不是实时行情。
- 行业和关系证据用于支持研究范围、经济机制和假设，不能直接写成已确认客户、供应商或合同事实。
- 当前向量数据库路径以常驻连接和 CPU 索引为主，尚未把 GPU 向量索引作为主链路能力。
- 行业深度和供应链传导问题仍然耗时较高，对缓存状态和模型服务延迟敏感。
- 工作台适合本地开发、演示和诊断评测；多用户生产服务还需要权限、队列、数据库、成本控制和更完整的并发治理。
- 校验器能阻止越界和缺证结论，但高质量研究仍取决于上游数据覆盖、关系证据质量和专家结论卡密度。
